# Devnet end-to-end test — 2026-05-13

First real end-to-end run of the chain settlement path against a live xion devnet (`xion-devnet-1`, wasmd v0.54.0, xiond v20.0.0). Until this run, the contract had only been exercised under `cw-multi-test` (which bypasses the JSON wire boundary), and the Go client had only unit tests with stubbed responses. v0.3.1 was an audit-driven fix release, not a real-chain validation.

## Result

End-to-end round trip works once you route around six bugs in the deploy/sync tooling. The contract itself is correct: every v0.3.1 audit fix verified against a live chain (string-encoded numerics, decimal-nanosecond timestamps, dual-canonical design, base64 pubkey/signature encoding, reward math). Final on-chain state:

```
adversary-alpha: balance=116 tp=1 fp=0
  finding F-e2e-001 (critical, stake 8) → resolution true_positive (stake_returned 8, reward 16)
```

Math checks out: 100 (initial) − 8 (commit stake) + 8 (return) + 16 (reward = stake × outcome_reward_multiplier 2) = 116.

## Findings for v0.3.2

### F1 — `--optimize` is required on xion devnet, not optional

`scripts/deploy-contract.sh` documents `--optimize` as optional, "without it the dev-built wasm is uploaded as-is." But wasmd v0.54.0 on xion-devnet-1 rejects the dev build:

```
Error during static Wasm validation: Wasm bytecode could not be deserialized.
Deserialization error: "bulk memory support is not enabled (at offset 0xaec)"
```

The unoptimized wasm contains bulk-memory ops that the chain refuses to deserialize. `RUSTFLAGS="-C target-feature=-bulk-memory,..."` does not help — the ops come from a downstream dep (probably `cosmwasm-std`'s panic glue or the wasm-stripping pass the optimizer runs).

**Fix:** make `--optimize` the default in the script, with a `--skip-optimize` opt-out for environments that don't have docker. Update the inline docs.

### F2 — `cosmwasm/optimizer:0.16.0` is broken by a transitive dep bump

The pinned optimizer image ships Rust 1.78.0 / Cargo 1.78. `base64ct v1.8.3` (transitive through `cosmwasm-std`) declares `edition2024`, which requires Cargo 1.85+. Build fails immediately:

```
error: failed to parse manifest at .../base64ct-1.8.3/Cargo.toml
Caused by: feature `edition2024` is required
```

`cosmwasm/optimizer:0.17.0` works (verified — produces a clean 425 KB stripped wasm).

**Fix:** bump the pinned tag in `deploy-contract.sh` from `0.16.0` to `0.17.0` (or pin via env var). Document the upgrade in CHANGELOG.

### F3 — `tribunal chain init` accepts `tcp://` but `chain status` rejects it

`xiond` uses `tcp://...:26657` as the RPC URL convention; Go's `net/http` requires `http://` or `https://`. The `chain init` command happily writes `tcp://...` to `chain.yaml`, then the next command fails with:

```
Error: status http: Get "tcp://localhost:26657/status": unsupported protocol scheme "tcp"
```

**Fix:** in `chain init`, auto-translate `tcp://` → `http://` (with a warning) or reject it at write time with a clear error. The xiond binary takes `tcp://` because that's the Tendermint convention; the Go HTTP client never does.

### F4 — Account sequence mismatch on back-to-back txs

`tribunal chain register foo && tribunal chain register bar` (or `register` immediately followed by `sync`) fails on the second tx with:

```
account sequence mismatch, expected 4, got 3
```

The first tx broadcasts and returns success, but the second tx fires before the first has been included in a block, so xiond's cached account sequence is one behind. The `chain sync` command's _internal_ commit→resolve pipeline hits the same bug because both batches are dispatched without waiting for inclusion.

**Fix:** the chain client (`internal/chain/client.go`) needs a "wait for inclusion" hook after `Execute` — either poll `/tx?hash=...` until the tx is found, or use `--broadcast-mode block` (deprecated in newer xiond but cleaner). Sync orchestration must use this between batches.

### F5 — `chain sync` cannot recover from a partial failure

When the resolution half of a sync fails (e.g. because of F4's sequence mismatch), the commit half has already landed on-chain. Re-running sync re-attempts the full ledger:

```
finding P-e2e-001/F-e2e-001 already committed
```

There's no local sync-state cache, so every retry permanently fails until the user manually intervenes.

**Fix:** before building a `commit_finding_batch`, query the chain for each finding's state and filter out ones already committed. Or: have sync write a `.tribunal/sync-state.json` of last-committed entries. The first option is more correct (handles out-of-band commits); the second is faster.

### F6 — `outcome_reward_multiplier` in `chain.yaml` is local-only and unsynced

`chain init` doesn't query the contract for the deployed `outcome_reward_multiplier`, so `chain.yaml` ends up with `0` even when the contract has it set to 2. This field is described as "mirrors the contract's instantiation parameter for client-side reward previews" — but it lies if the user doesn't manually edit the yaml.

**Fix:** in `chain init`, after writing the basics, fetch the contract `config` and populate `outcome_reward_multiplier`. Or remove the field entirely and have any client preview re-query.

## Wins (v0.3.1 audit fixes verified)

Everything the v0.3.1 audit packet fixed was confirmed correct end-to-end:

- **String-encoded numerics** (`balance`, `stake`, `stake_returned`, `reward`, `initial_balance`, `rotation_floor`, `outcome_reward_multiplier`) round-trip cleanly between Rust `Uint128`-as-decimal-string and Go `string` types.
- **Decimal-nanosecond timestamps** (`created_at`, `committed_at`, `resolved_at`) deserialize correctly on the Go side as strings.
- **`AgentResp` includes `pubkey`** and `ReputationResp` exposes the agent record as expected.
- **`ResolutionRecord` split** into `stake_returned` + `reward` works — `tribunal chain query finding` shows both.
- **Dual-canonical signing** (ledger signs over JSON-canonical, chain signs over pipe-canonical via `BuildFindingCommit` / `BuildResolutionCommit` re-signing) — the contract verifies both halves correctly.
- **Reward math** — stake (8) × outcome_reward_multiplier (2) = 16, applied on true_positive resolution.
- **tp_count / fp_count** align as uint64 on both sides.

## v0.3.2 scope

All six findings above are tooling/operator-experience defects, not contract-correctness defects. The contract itself shipped clean in v0.3.1. v0.3.2 should be scoped as:

- F1, F2 → `scripts/deploy-contract.sh` updates (default `--optimize`, bump optimizer tag)
- F3, F6 → `tribunal chain init` polish (URL scheme normalization, contract-query population)
- F4 → `internal/chain/client.go` wait-for-inclusion helper
- F5 → `tribunal chain sync` idempotency (pre-flight chain-state filter)

Estimated scope: <300 LOC across Go + bash. No contract changes needed; no chain migration required. Pure tooling release.
